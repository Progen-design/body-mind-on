# AI Product Guardrails – Body & Mind ON

> Technical product safety documentation.  
> This is NOT legal advice. It does not constitute legal compliance or professional medical/nutritional guidance.

---

## 1. Nature of AI-Generated Outputs

### Fitness & Wellness Recommendations Only
All outputs generated by the AI trainer agent (meal plans, training plans, supplementation, mindset sections) are:
- **Fitness and wellness recommendations**, not medical advice
- Not a substitute for consultation with a licensed physician, dietitian, or certified personal trainer
- Tailored to the user's stated profile data (weight, goal, dietary preferences, activity level)

The platform does not diagnose, treat, cure, or prevent any disease.

### User Profile as Input
The AI system reacts to user-provided data:
- Body metrics (height, weight, age, gender)
- Stated goals (fat loss, muscle gain, maintenance)
- Dietary preferences and restrictions
- Activity level and stress level
- Selected habits

The accuracy of outputs depends directly on the accuracy of user-provided input. The system does not verify medical conditions.

---

## 2. Autonomous AI Behavior

### What the System Does Autonomously
- Generates a personalized weekly plan upon registration
- Sends the plan via email after successful generation
- Generates a coach onboarding message upon registration
- Reacts to user events (preferences change, progress check-in) by scheduling new tasks
- Adjusts plans based on user state changes through the task pipeline

### What the System Does NOT Do Autonomously
- Does not modify medication or supplement dosing based on medical conditions
- Does not override user-set dietary restrictions without explicit user update
- Does not generate content claiming medical efficacy

### Failsafe: Plan Always Delivered
The system uses a "no plan nowhere" principle: even if validation agents flag issues, the trainer still publishes a plan with a validation warning stored internally. The user always receives some plan output.

---

## 3. Third-Party Processors and APIs

The following external services are used:

| Service | Purpose |
|---|---|
| OpenAI (GPT-4.1 / mini) | AI plan generation, coaching messages, content drafts |
| Supabase | Database, authentication, user data storage |
| Vercel | Hosting, serverless functions, cron jobs |
| Spoonacular / RapidAPI | Meal image metadata and enrichment |
| ExerciseDB / RapidAPI | Exercise GIF and metadata enrichment |
| Pexels | Image fallback for meals and exercises |
| Nodemailer / Gmail SMTP | Email delivery of plans |
| Stripe | Membership and payment processing |

User data (body metrics, goals, dietary preferences) is transmitted to OpenAI as part of plan generation prompts.

---

## 4. Content Review

### Generated Plan Content
- AI-generated plans are not reviewed by a human before delivery to the user
- Nutrition and training validators run automatically before publication
- Plans with validation warnings are still published (production: logged internally)

### Marketing and Social Content
- `ai_content_drafts` created by marketing/social agents have `status: 'draft'` by default
- These drafts require admin review before any external publication
- Do not publish AI content drafts to social media or marketing channels without review

---

## 5. Data Retention and User Rights

- User profile data is stored in Supabase under the user's account
- Users can request account deletion via the profile settings page (`/api/delete-account`)
- Generated plans are linked to `user_id` and removed on account deletion
- AI task logs (`ai_logs`) are retained for operational debugging (recommended: 90-day retention)

---

## 6. System Limitations

- AI outputs may contain inaccuracies, especially for rare dietary restrictions or highly specific medical conditions
- Image enrichment (meals, exercises) depends on external API availability and relevance
- Plan generation may fail if OpenAI API is unavailable; tasks are retried with exponential backoff
- The system does not track caloric intake or exercise completion – it only generates recommendations

---

## 7. Operational Responsibility

The operator (Body & Mind ON) is responsible for:
- Ensuring the platform is not used for medical diagnosis
- Monitoring AI task failures and DLQ (dead letter queue) entries
- Reviewing marketing/social drafts before external use
- Applying Supabase migrations before deploying code changes
- Keeping API keys secure and rotating them per security best practices
